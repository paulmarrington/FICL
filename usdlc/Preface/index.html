Preface

Before uSDLC can help us with the design we need to have a high level concept on how our project will look. Without this our design process lacks direction. There is a process of consideration of different options and decisions that need to be made first. If we document the process here they provide a starting point for the design and a perspective on the decisions made for a clearer understanding of the project.

Reason - Result - User Interface - Language - Frameworks - Deployment

Reason - notes of why this project is needed

When developing an Android application I had the need for user's to be able to configure the operation of the application based on a diverse set of criteria and information. It was impossible for me to document all needed cases as the users of the resulting system do not know themselves - until they start using the final product. Iterative/agile development relieves this sort of problem, but a traditional software approach would leave my clients with a system 'set in stone'.

To provide the flexibility I wanted I needed to be able to send templates from the server to the phone to make workflow decisions.

Investigation - the path to a decision

I decided that the most flexible solution would be a scripting language that could be sent from the server and refer to application data to set other data elements - such as turning gps tracking on or off.

Android is Java based. There are quite a few scripting languages that run on the JVM - including Groovy, Jython, jRuby, Javascript and Coffeescript. The problem is that they are all big. They make your application huge and are all known to run slowly, if at all, on Android. To be fair, they are not designed for configuration and workflow - they are intended as general computer languages.

Another consideration was operating environment. This application, in common with most, will have an iPhone clone. I needed a scripting language that would run on both.

BASIC and it's ilk are relatively lightweight. I could not find a consistent dialect that ran on Java and Objective-C.

Result - a high level description of what the finished project will look like

I decided to write one. By making a FORTH-like stack machine it would be much smaller than even BASIC, easier to develop, faster and more extensible.

Most existing FORTH implementations are for embedded systems, typically including a built-in assembler for fast native code. My requirement was application support where the application code can do all the heavy lifting with the control language acting as glue.

I called if FICL (Fast Interactive Configuration Language) to clearly show that it is not a general programming language. It sacrifices readability for speed and size. It is up to a server based driving system to create the commands based on templates and simplified user input.

By keeping the core simple, I plan to create a C version using the Java version as a template. I can then write wrappers in Objective-C and C# so that the same code supports iPhone, iPad and Windows Phone.

User Interface - technology, look and feel, user experience

FICL is designed to be an application to application facility, so the user is directly another computer and indirectly a script writer who creates the templates.

For my application I will be using push technology to send workflow changes on demand from a web server.

Language - computer languages and libraries decided upon

Android uses Java as the primary application development language. By writing FICL in Java, it will be available to the plethoria of other JVM based language as needed.

Android applications support C code. The problem is that any pure java application would have to move do a mixed application to make use of C. So added complexity in development and deployment outweighs any slight performance hit using the Dalvik JVM.

No such choice exists for iOS devices. Here we have C, C++ and Objective C. Since C++ is an orphan in this environment, I decided to develop the core in C and add an Objective C wrapper so that FICL can access object methods and fields. The downside is that C does not natively support dynamic size arrays or strings, dictionaries, ARC or reflection to call native code.

The choice to write it in C rather than Objective C is more fickle than FICL. A C version can be used almost anywhere a Java version cannot - Cocoa on iOS and OS X, .NET on Windows Phone, Windows desktop and Windows Server, as a native library for Python and on embedded devices that have a C cross-compiler.

Frameworks - hardware and software frameworks anticipated

  • J2SE applications (command line, GUI, Webstart, applet)
  • J2EE servers (Tomcat, Websphere, ...)
  • Groovy applications
  • Grails servers
  • Iphone and iPad
  • Windows Mobile
  • Android
  • embedded devices with C cross-compilers

Deployment - envisioned deployment environment

  • JVM: A single jar with 1 class and 2 resource files.
  • iOS: Dynalib
  • OS X: Dynalib
  • .NET: DLL
